# Store artifacts verbatim; gate raw-log access rather than scrub at rest

Calculation artifacts (electronic-structure log files) are stored byte-for-byte and content-addressed by SHA-256; the stored bytes are never rewritten. Redacting personally-identifying or infrastructure strings — scratch paths, usernames, cluster hostnames, scheduler job IDs — at upload would break content-addressing, because the hash would no longer match the producer's original file, and with it the end-to-end, re-runnable traceability the four-role model promises. Scrubbing is therefore never applied to bytes at rest, and contributor-side scrubbing is not relied upon.

The exposure those strings create lives only in the raw artifacts, not in the queryable science: parsed products — energies, geometries, levels of theory, and the execution-parameter vocabulary — are derived by extracting scientific content and carry no scratch paths or identities. That parsed layer is the public face of the database. Raw-artifact download is therefore an access-controlled operation, served only to authorized callers and gated by review status, so that unredacted logs never reach anonymous clients; retrieval of raw bytes remains verifiable against the stored SHA-256.

Where raw logs must be exposed beyond authorized callers, redaction happens at egress, never at rest: a scrub-on-download mode returns a clearly-labelled redacted copy whose bytes intentionally differ from the stored hash, rather than the verbatim artifact. This is an explicit, opt-in egress transform and never the default, because heuristic redaction is lossy and cannot be trusted to catch every identifier; it supplements, and never substitutes for, access control at the database boundary.
